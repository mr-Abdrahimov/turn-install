# Обход капчи VK «Я Не Робот» (not_robot_captcha)

## Суть проблемы

vk-turn client добывает TURN-креды, авторизуясь в VK по ссылке на звонок. VK защищает эту
авторизацию капчей **not_robot** (`error_code:14`, `redirect_uri` + `session_token`; старые поля
`captcha_sid`/`captcha_img` больше не присылаются).

Два независимых препятствия:

1. **Баг парсера в релизе v1.8.3.** `ParseVkCaptchaError` (client/main.go) делал `return nil`,
   если нет `captcha_sid`/`captcha_img`, и НЕ доходил до встроенного решателя капчи — хотя тому
   нужны только `redirect_uri` + `session_token`. → Патч `patches/vk-captcha-not-robot.patch`
   делает эти поля необязательными (собери клиент через `build-client.sh`).

2. **Сама капча решается только настоящим браузером.** После патча решатель запускается, но:
   - `auto` → VK отвечает `status=BOT`;
   - `slider POC` → `BOT/ERROR`;
   - `manual` → открывает локальную страницу капчи и ждёт, пока человек пройдёт чекбокс в браузере.

   Полностью автоматического headless-обхода на роутере сейчас нет — это ограничение VK, не бага.

vk-turn client при неудаче авто-решения поднимает локальный веб-сервер капчи на
**`127.0.0.1:8765`** роутера и пишет в лог `ACTION REQUIRED: MANUAL CAPTCHA SOLVING NEEDED`.
Патч поднял таймаут ожидания до **10 минут**.

## Способ 1 (рекомендуется) — через страницу роутера, БЕЗ ssh

На роутере работает **LAN-шлюз капчи** (`captcha-lan-gw`, procd-сервис `vkturn-captcha-gw`,
порт **8766**). Он проксирует страницу капчи на LAN и переписывает `localhost:8765` на адрес
роутера, поэтому её можно открыть с любого устройства в сети:

1. Дождись в логе `ACTION REQUIRED` (или просто открой страницу — она сама покажет, когда готово):
   ```sh
   ssh root@192.168.1.1 'logread -e vkturn | tail'
   ```
2. С любого устройства в LAN открой в браузере:
   ```
   http://192.168.1.1:8766
   ```
   (замени на IP своего роутера). Пока капча не нужна — страница покажет «Капча ещё не готова».
3. Реши капчу VK «Я не робот».
4. Клиент подхватит токен → `Success with client_id=…` и `Established DTLS connection!`.

Так **любой пользователь** может решить капчу сам, без знания ssh-проброса портов.

## Способ 2 (запасной) — ssh-проброс порта

Если LAN-шлюз недоступен:

1. Проброс порта капчи к себе (окно держать открытым):
   ```sh
   ssh -L 8765:localhost:8765 root@192.168.1.1
   ```
2. В браузере открой `http://localhost:8765` и реши капчу.

Если не успел за 10 минут — клиент перезапустится (procd) и снова откроет капчу
(страница `http://192.168.1.1:8766` продолжит работать, просто обнови её).

## После успешной авторизации

Подними AmneziaWG-туннель (если `auto=1` — поднимется сам, как только клиент установит DTLS):
```sh
ssh root@192.168.1.1 'ifup VKTURN; sleep 5; awg show VKTURN | grep -E "handshake|transfer"'
```
`latest handshake: N seconds ago` + растущий `transfer` = туннель работает.

## Ограничения и эксплуатация

- **Капчу нужно решать при каждой (пере)авторизации** в VK: после перезагрузки роутера,
  перезапуска клиента или протухания VK-сессии. Пока DTLS-стримы живут — туннель держится без
  вмешательства.
- Ссылка на звонок VK тоже протухает: обнови через `vkturn set-link https://vk.ru/call/join/…`.
- Проверка внешнего IP через туннель:
  ```sh
  ssh root@192.168.1.1 'ip route add 104.26.12.205 dev VKTURN src 10.8.1.2;
    curl -s --resolve api.ipify.org:443:104.26.12.205 https://api.ipify.org; echo;
    ip route del 104.26.12.205 dev VKTURN'
  # должен вернуть IP сервера
  ```
